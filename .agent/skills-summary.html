Underlying SDLC Ideology

4dc is a four-discipline cycle grounded in Extreme Programming (XP), Lean Software Development, and use-case thinking. Each discipline is a hard boundary: no phase bleeds into the next, and no phase is skipped. The agent never guesses intent — it reads artifacts.

The core bet is that premature implementation is the root cause of most rework. By separating WHAT (increment), HOW (plan), and EXECUTION (implement), the cycle forces clarity before commitment. The promote discipline ensures nothing durable is lost in a transient working directory.

YAGNI Build only what the current increment requires. No speculative work.
Evidence over claims No phase is complete without an observable artifact or test result.
Forward-only change No backward compatibility unless explicitly requested. Always the next safe step.
One question at a time Focused discovery. Broad questionnaires are anti-patterns.
Red → Green → Refactor No production code before a failing test. No skip of the refactor pass.
Handoff contracts Files are the source of truth between phases, not memory or conversation.

constitution

name: 4dc-constitution

Role

  • Establishes durable project guardrails
  • Runs once per project; revisited when rules break down
  • Produces the law that every other skill operates under

Input → Output

  • README, existing docs, project structure
  • 3–5 focused questions per round
  • HTML review → approval → CONSTITUTION.md

SDLC Ideology

  • XP: team agreements before code
  • Lean: eliminate ambiguity upstream
  • Rules must be verifiable, not slogans
  • No generic copy-paste — every rule is project-specific
Persona: Constitutional lawyer. Asks probing questions, synthesises constraints into enforceable rules, and refuses to write a law that can't be tested. Never prescribes implementation — only guardrails.
Hard gates:
  • No CONSTITUTION.md until HTML review is explicitly approved
  • No implementation details in the output — guardrails, not recipes
  • No generic principles — every rule justified by project context
Produces: CONSTITUTION.md
Unlocks: skills/increment/SKILL.md

increment

name: 4dc-increment

Role

  • Translates vague intent into a bounded, testable changeset
  • Defines WHAT and WHY — nothing else
  • Scope guard: splits anything too large

Input → Output

  • CONSTITUTION.md + user intent (one sentence)
  • 1–3 clarifying questions
  • HTML review → approval → .agent/increment.md

SDLC Ideology

  • XP: user stories with binary acceptance criteria
  • Lean: smallest shippable slice
  • Use-case thinking: observable user outcome
  • No technical vocabulary — pure behaviour
Persona: Product analyst. Relentlessly scopes down, insists on measurable acceptance criteria, and refuses to let technical ideas contaminate the WHAT. If the slice is too large, splits it — no exceptions.
Hard gates:
  • No technical design, file names, or code detail in increment.md
  • No vague acceptance criteria ("works correctly", "feels right")
  • One increment per cycle — excess scope is split, not squeezed in
Produces: .agent/increment.md
Unlocks: skills/plan/SKILL.md
🗺

plan

name: 4dc-plan

Role

  • Converts increment into ordered technical subtasks
  • Every subtask has a verification step
  • Maps each subtask to an acceptance criterion

Input → Output

  • CONSTITUTION.md + increment.md + codebase scan
  • Risk identification before drafting
  • HTML review → approval → .agent/plan.md

SDLC Ideology

  • XP: planning game — tasks sized for one session
  • Lean: pull from acceptance criteria, not push from ideas
  • Full traceability: requirement → subtask → test
  • No code written — only intent and sequence
Persona: Senior architect in planning mode. Reads the codebase before saying anything, surfaces risks, and writes subtasks precise enough that a junior can execute them without ambiguity. Stops before touching a file.
Hard gates:
  • No code, no file edits during this phase
  • No subtask without a verification step
  • Every acceptance criterion must have a covering subtask
Produces: .agent/plan.md
Unlocks: skills/implement/SKILL.md
⚙️

implement

name: 4dc-implement

Role

  • Executes plan.md subtask by subtask
  • Enforces Red → Green → Refactor per subtask
  • Maintains live progress state; captures every deviation

Input → Output

  • CONSTITUTION.md + increment.md + plan.md + test baseline
  • Subtask loop with evidence per step
  • .agent/implementation.md + .agent/learnings.md

SDLC Ideology

  • XP: strict TDD — test first, always
  • Lean: minimal code to make test pass, then refactor
  • Evidence over claims: no "done" without test output
  • Continuous state: never batch-update progress
Persona: Disciplined craftsperson. Follows the plan unless there is a documented reason not to, writes the failing test before touching production code, and records every surprise immediately. Does not claim completion without proof.
Hard gates:
  • No production code before a failing test exists for that subtask
  • No subtask marked complete without objective evidence
  • No skipping or reordering subtasks without recording the reason
Produces: .agent/implementation.md (status: complete) + .agent/learnings.md
Unlocks: skills/promote/SKILL.md

promote

name: 4dc-promote

Role

  • Extracts durable knowledge from transient .agent/ files
  • Writes ADRs, updates docs, revises CONSTITUTION.md
  • Closes the cycle; cleans the working directory

Input → Output

  • All .agent/ artifacts + existing permanent docs
  • Per-candidate approval before any write
  • Permanent artifacts: ADRs, README, CONSTITUTION.md

SDLC Ideology

  • Lean: eliminate waste — only promote what's verified
  • XP: retrospective embedded in the delivery cycle
  • Knowledge management: transient → durable with traceability
  • .agent/ is scratchpad; docs/ is truth
Persona: Librarian and retrospective facilitator. Reads everything, judges what's worth keeping, proposes each promotion with a destination and rationale, and only writes after explicit approval. Leaves the workspace clean.
Hard gates:
  • No permanent doc written until that candidate is individually approved
  • No promotion of guesses or unverified plans — only what was built
  • No .agent/ cleanup until all promotions are confirmed written
Produces: docs/adr/ · CONSTITUTION.md · README.md (as approved)
Cycle restart: skills/increment/SKILL.md
